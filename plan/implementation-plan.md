# 盘点工具 1.0 实施计划

## 1. 项目目标

建设一套盘点工具 1.0，覆盖总部发起月盘任务、门店小程序按分区录入、系统自动汇总、提交后更新门店默认分区物料清单的完整闭环。

技术栈：

- 后端：Spring Boot
- 管理端：Vue 3
- 小程序端：原生微信小程序

核心业务闭环：

```text
基础系统同步物料和门店
-> 总部维护盘点模板
-> 总部创建月盘任务
-> 系统生成任务快照
-> 店长按分区录入
-> 系统汇总跨分区物料
-> 店长提交
-> 系统锁定任务并更新门店默认清单
-> 门店/总部查看结果
```

## 2. 推荐工程结构

```text
盘点工具/
  backend/                    # Spring Boot 后端
  admin-web/                  # Vue 3 总部管理端
  miniprogram/                # 原生微信小程序
  docs/                       # 数据库、接口、业务规则文档
  plan/                       # 项目执行计划
  PRD/                        # 产品需求文档
  design/                     # 设计稿
```

## 3. 关键业务规则

### 3.1 任务规则

- 月盘任务由总部创建，门店不能自行创建任务。
- 任务状态包括：
  - `not_started`：未开始
  - `in_progress`：进行中
  - `submitted`：已提交
- 已提交任务不可编辑、不可重复提交。
- 创建任务时生成任务快照。
- 任务快照生成后，不受后续模板或门店默认清单变更影响。

### 3.2 分区规则

- 分区是盘点执行单位。
- 每个分区维护独立物料列表和显示顺序。
- 门店可在当前任务中调整分区与物料结构。
- 门店可添加分区、删除空分区、添加物料、调整未录入物料顺序。
- 已录入物料不可移动、不可删除。

### 3.3 物料规则

- 物料主数据由基础数据库管理系统维护。
- 盘点工具只同步并引用物料，不新增、不编辑、不删除物料。
- 门店不可新建基础数据库管理系统不存在的物料。
- 同一物料允许出现在同一任务的多个分区。
- 同一分区内不可重复存在同一物料。

### 3.4 录入规则

- 数量允许整数、小数和 0。
- 数量不允许负数。
- 空值表示未录入。
- 0 表示已录入为 0，必须和空值严格区分。
- 备注非必填。
- 单位固定为最小规格单位，门店不可修改。

物料录入状态：

- `not_entered`：未录入
- `entered`：已录入且数量大于 0
- `zero_entered`：已录入且数量等于 0

### 3.5 汇总规则

- 系统按任务维度汇总相同物料在多个分区中的录入数量。
- 汇总公式：同一任务下同一物料的所有分区录入数量求和。
- 空值不参与汇总。
- 0 参与汇总。

### 3.6 提交规则

- 仅店长可提交任务。
- 所有分区内所有物料均完成录入后才可提交。
- 任一物料数量为空时不可提交。
- 提交后任务锁定。

### 3.7 默认清单回写规则

- 系统为每个门店维护一套门店默认分区物料清单。
- 该清单用于后续任务生成。
- 仅在任务提交成功后更新门店默认清单。
- 若某条分区物料录入数量为 0，则从门店默认清单中移除该分区物料关系。
- 移除默认清单关系不影响当前任务历史数据、不删除总部模板、不删除物料主数据。

目标：建立三端工程骨架和统一开发规范。

验收标准：

- 三端工程能够启动。

- 小程序具备基础页面结构。

## 5. 阶段 1：后端数据库与核心模型

目标：先把业务数据结构定稳。

核心表：

```text
material                      # 物料同步缓存
inventory_template            # 盘点模板
template_zone                 # 模板分区
template_zone_material        # 模板分区物料关系
store                         # 门店同步缓存
store_zone_material           # 门店默认分区物料清单
inventory_task                # 月盘任务
task_zone                     # 任务分区快照
task_zone_material            # 任务分区物料快照
task_material_summary         # 任务物料汇总
user_account                  # 用户账号
```

关键设计要求：

- `task_zone_material.input_qty` 必须允许 `null`。
- `null` 表示未录入。
- `0` 表示已录入为 0。
- 任务快照表要保存物料名称、规格、单位快照字段。
- 排序字段统一使用 `sort_no`。
- 建议所有核心表包含 `created_at`、`updated_at`。
- 需要逻辑启停的表使用 `status`。
- `material` 和 `store` 需要保存外部系统 ID、来源系统、同步状态和最近同步时间。

验收标准：

- 数据库 migration 可重复执行。
- 表结构覆盖 PRD 核心模型。
- 初始化数据包含门店、用户、物料、模板、默认清单。
- 实体类、Mapper/Repository、基础枚举完成。

## 6. 阶段 2：后端基础管理能力

目标：让总部可以查询基础主数据，并维护盘点模板。

### 6.1 主数据查询 API

```text
GET    /api/admin/materials
GET    /api/admin/materials/{id}
GET    /api/admin/stores
GET    /api/admin/stores/{id}
```

功能：

- 物料列表
- 物料搜索
- 门店列表
- 门店搜索
- 查看同步状态

验收标准：

- 可以查询同步缓存中的物料和门店。
- 不在盘点工具中新增、编辑、启停物料和门店。
- 停用物料不可被新添加到模板或任务分区。
- 停用门店不可创建新的月盘任务。

### 6.2 模板 API

```text
GET    /api/admin/templates
POST   /api/admin/templates
GET    /api/admin/templates/{id}
PUT    /api/admin/templates/{id}
PATCH  /api/admin/templates/{id}/status
POST   /api/admin/templates/{id}/zones
PUT    /api/admin/templates/{id}/zones/{zoneId}
DELETE /api/admin/templates/{id}/zones/{zoneId}
```

### 6.3 模板分区物料 API

```text
POST   /api/admin/template-zones/{zoneId}/materials
DELETE /api/admin/template-zone-materials/{id}
PUT    /api/admin/template-zones/{zoneId}/materials/sort
```

关键规则：

- 同一模板分区内不可重复添加同一物料。
- 同一物料允许出现在模板的多个分区。
- 删除分区时处理其下物料关系。
- 排序使用 `sort_no`。

验收标准：

- 可创建模板。
- 可添加分区。
- 可给分区添加物料。
- 可删除分区物料。
- 可调整物料顺序。
- 重复添加会被后端拦截。

## 7. 阶段 3：后端月盘任务与快照

目标：实现总部创建任务，并生成独立任务快照。

后台 API：

```text
GET  /api/admin/inventory-tasks
POST /api/admin/inventory-tasks
GET  /api/admin/inventory-tasks/{id}
GET  /api/admin/inventory-tasks/{id}/result
```

创建任务流程：

1. 校验门店存在。
2. 校验模板存在且可用。
3. 创建 `inventory_task`。
4. 基于模板和门店默认清单生成 `task_zone`。
5. 生成 `task_zone_material`。
6. 保存物料名称、规格、单位快照。
7. 初始化 `input_qty = null`。
8. 初始化 `input_status = not_entered`。
9. 初始化任务状态为 `not_started`。

实现注意：

- 任务快照和模板必须解耦。
- 后续修改模板不影响已创建任务。
- 后续修改门店默认清单不影响已创建任务。
- 任务列表进度建议实时计算。

验收标准：

- 创建任务后能看到任务分区快照。
- 修改模板后，已创建任务不变化。
- 任务初始进度正确。
- 任务列表能显示门店、月份、模板、截止时间、状态、分区数、物料数。

## 8. 阶段 4：小程序任务与录入闭环 API

目标：支撑店长完成盘点主流程。

门店端 API：

```text
GET  /api/store/inventory-tasks
GET  /api/store/inventory-tasks/{id}
GET  /api/store/inventory-tasks/{id}/zones/{zoneId}
PUT  /api/store/inventory-tasks/{id}/zones/{zoneId}/inputs
GET  /api/store/inventory-tasks/{id}/summary
POST /api/store/inventory-tasks/{id}/submit
GET  /api/store/inventory-tasks/{id}/result
```

保存分区录入规则：

- 数量允许整数、小数、0。
- 数量不允许负数。
- 空值表示未录入。
- `input_qty > 0` 时，状态为 `entered`。
- `input_qty = 0` 时，状态为 `zero_entered`。
- `input_qty = null` 时，状态为 `not_entered`。
- 已提交任务不可保存。

验收标准：

- 小程序可以查询任务列表。
- 可以进入任务详情。
- 可以查询分区物料。
- 可以保存分区盘点数据。
- 任务进度实时正确。
- 0 和空值严格区分。

## 9. 阶段 5：提交事务与汇总

目标：实现最核心业务闭环。

提交接口：

```text
POST /api/store/inventory-tasks/{id}/submit
```

提交逻辑必须放在后端事务中：

1. 校验当前用户为店长。
2. 校验任务存在。
3. 校验任务不是 `submitted`。
4. 校验所有 `task_zone_material.input_qty` 都不为 `null`。
5. 按 `task_id + material_id` 汇总数量。
6. 清空并重建 `task_material_summary`。
7. 更新任务状态为 `submitted`。
8. 写入 `submitted_by`。
9. 写入 `submitted_at`。
10. 更新门店默认分区物料清单。
11. 对录入数量为 0 的分区物料关系，从默认清单中移除。
12. 保留当前任务快照历史不变。

验收标准：

- 未录完不能提交。
- 录入 0 可以提交。
- 跨分区同物料汇总正确。
- 提交后任务不可编辑。
- 提交后默认清单正确更新。
- 历史任务数据不变。

## 11. 阶段 7：原生微信小程序

目标：实现门店端盘点体验。

页面结构：

```text
pages/task-list/index              # 月盘任务列表
pages/task-detail/index            # 任务详情
pages/zone-input/index             # 分区盘点
pages/zone-adjust/index            # 调整分区与物料
pages/add-zone/index               # 添加分区
pages/add-material/index           # 添加物料
pages/task-material-search/index   # 任务内物料搜索
pages/summary/index                # 汇总预览
pages/result/index                 # 盘点结果
```

优先实现主链路：

1. 任务列表。
2. 任务详情。
3. 分区盘点。
4. 保存分区。
5. 汇总预览。
6. 提交任务。
7. 查看结果。

再实现增强链路：

1. 调整分区与物料。
2. 添加分区。
3. 添加物料。
4. 当前分区搜索。
5. 搜索未命中兜底。
6. 任务内物料加入当前分区。
7. 物料库加入当前分区。

小程序端重点：

- 输入框要严格处理空值、0、小数、负数。
- 当前分区搜索要实时过滤。
- 当前分区无结果时显示兜底入口。
- 已录入物料不可移动、不可删除。
- 已提交任务全部只读。
- 底部主按钮根据任务状态动态变化。

验收标准：

- 店长能完成一次完整盘点。
- 未录入不能提交。
- 提交后所有页面只读。
- 搜索、添加、调整符合 PRD 限制。

## 15. 第一轮开发建议

第一轮只做最小可用闭环：

1. 后端建表和 seed 数据。
2. 后端创建任务并生成快照。
